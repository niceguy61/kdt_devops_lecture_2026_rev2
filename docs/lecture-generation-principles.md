# 강의 생성 원칙

## 목적
이 문서는 `docs/plan.md`의 주차별 커리큘럼을 실제 강의 자료로 만들 때 적용할 공통 원칙이다.

강의의 목표는 단순히 Docker, Kubernetes, AWS, Terraform 명령어를 가르치는 것이 아니다. 주니어 인프라/클라우드 엔지니어가 개발팀과 협업하며 서비스를 안정적으로 실행, 배포, 관찰, 개선할 수 있도록 사고방식과 운영 기준을 만드는 것이다.

## 강의 작성 기본 관점
- 개발자가 아니라 인프라/DevOps 엔지니어를 키운다는 관점으로 설명한다.
- "어떻게 개발하는가"보다 "어떻게 실행하고, 연결하고, 배포하고, 관찰하고, 장애를 줄일 것인가"를 중심에 둔다.
- 도구의 기능보다 도구가 해결하려는 운영 문제를 먼저 설명한다.
- 학생이 명령어를 외우게 하지 말고, 어떤 상태를 확인하려는 명령인지 이해하게 한다.
- 강의 자료는 가능한 한 공식 문서, 공식 블로그, 공식 발표 자료를 근거로 구성한다.
- 매 수업은 개념, 비유, 실습, 관찰, 정리의 흐름을 가진다.

## 공식 자료 우선 원칙
강의안에는 공식 자료를 적극적으로 포함한다. 학생이 AI 답변이나 검색 결과만 믿지 않고, 실제 현업에서 기준이 되는 문서를 찾아 읽고 검증하는 습관을 만들기 위함이다.

우선순위는 다음과 같다.

1. 서비스 또는 도구의 공식 documentation
   - 예: Docker Docs, Kubernetes Documentation, Helm Docs, Terraform Documentation, AWS Documentation

2. 클라우드/벤더 공식 블로그
   - 예: AWS Blog, Google Cloud Blog, Kubernetes Blog, CNCF Blog, HashiCorp Blog

3. 공식 글로벌 발표 자료와 컨퍼런스 세션
   - 예: AWS re:Invent, Google Cloud Next, KubeCon + CloudNativeCon, HashiConf

4. 공식 아키텍처 가이드와 Well-Architected 자료
   - 예: AWS Well-Architected Framework, AWS Architecture Center, CNCF Landscape, Kubernetes production best practices

강의 생성 시 각 주차 또는 주요 주제에는 가능한 한 다음을 포함한다.
- 공식 문서 링크
- 공식 문서에서 확인해야 할 키워드
- 문서를 읽을 때 주의할 전제 조건 또는 버전
- AI 답변과 공식 문서를 비교해 검증하는 질문

비공식 블로그, 커뮤니티 글, Stack Overflow, 개인 경험 기반 자료는 보조 자료로만 사용한다. 공식 문서와 충돌하면 공식 문서를 우선한다.

## 계약 기반 Tone And 운영 준수
강의안은 [Lecture Tone And 운영 준수 기준](./lecture-tone.md)을 따른다. 강의 경험과 현장 경력을 적절히 소개해 신뢰를 형성하되, 준비 부족이나 "처음"이라는 표현으로 신뢰를 낮추지 않는다.

강의안 작성 시 다음을 지킨다.
- 50분 수업과 10분 휴식, 점심시간, 교육 종료 시간을 존중한다.
- 학생과 운영자를 향한 경어와 존중 표현을 유지한다.
- 교육과 무관한 잡담, 개인 업무, 소속사/상품 홍보, 운영자/교보재/교육장 불평, 훈련생과의 언쟁 유도 표현을 넣지 않는다.
- 주교재와 강의 계획서를 중심으로 진행하고, 부교재나 외부 자료는 학습 목적과 출처가 분명할 때만 사용한다.
- 종료 전에는 당일 내용, 산출물, 다음 행동, feedback 경로를 정리한다.

## 90% 품질 Gate
강의안은 형식상 완성으로 통과하지 않는다. MIT 공대생 수준의 까다로운 학습자가 보아도 만족도 90% 이상을 기대할 수 있도록 설명 깊이, 공식/학술 근거, 현업 DevOps 판단, 시각 자료, 실습 가능성을 모두 갖춘다.

이 기준에 미달하면 작성 agent가 바로 pass를 선언하지 않는다. Retrieval Gate로 돌아가 공식 문서, 대학 강의자료, 공식 블로그, source pack을 다시 확인하고, 작성과 검토를 반복한다.

## 과밀 커리큘럼 대응 원칙
이 과정은 5주 안에 Docker, MSA, Kubernetes, AWS, Terraform/IaC까지 다루는 압축형 과정이다. 과밀하다는 이유로 필수 개념을 생략하거나 얕게 요약하지 않는다. 수업 시간 안에 모든 학생이 즉시 완전히 소화하지 못할 수 있음을 전제로, 교안은 수업 후 학생이 천천히 다시 읽고 따라올 수 있는 자습 자료까지 겸해야 한다.

강의 생성 agent는 자신이 생각하는 "필수 설명"의 100%만 쓰면 부족하다고 가정한다. 기본 답안보다 200% 확장한다는 기준으로 작성한다. 여기서 200% 확장은 불필요한 장황함이 아니라 다음을 추가한다는 뜻이다.

- 필수 개념의 정의, 등장 배경, 해결하는 운영 문제, 한계, 잘못 이해했을 때의 장애를 모두 설명한다.
- 각 개념마다 최소 하나 이상의 구체 예시, 명령 또는 콘솔 확인 지점, 기대 결과, 실패 증상, 주의할 점을 둔다.
- 학생이 나중에 혼자 따라올 수 있도록 시작 상태, 전제 조건, 경로, 버전, 입력값, 출력 예시, 정리 절차를 생략하지 않는다.
- "수업 중 핵심"과 "나중에 천천히 복습할 확장"을 구분한다. 필수 요소는 제목, 표, 체크리스트, 확인 기준에서 반복적으로 드러나야 한다.
- 과밀 주차에서는 모든 내용을 같은 비중으로 다루지 않는다. 필수 개념은 깊게, 선택/확장 개념은 언제 필요해지는지와 위험을 중심으로 설명한다.
- 짧은 bullet, 용어 나열, 명령어 목록만 있는 교안은 실패다. 학생이 문서만 읽어도 흐름을 복구할 수 있는 서술형 설명을 반드시 포함한다.

### 필수 개념 부각 규칙
각 lesson은 처음 10분 안에 `오늘 반드시 가져갈 것`을 명시한다. 이 섹션에는 최대 3~5개의 필수 개념만 넣고, 각 항목은 다음 형식을 갖춘다.

| 필수 개념 | 왜 필수인가 | 놓치면 생기는 문제 | 확인 지점 |
|---|---|---|---|

교안 본문에서는 필수 개념을 다시 설명할 때 같은 용어를 반복해도 된다. 단, 반복은 단순 복사가 아니라 "정의 -> 실습 -> 장애 -> 운영 판단 -> 산출물"처럼 다른 맥락에서 재등장해야 한다.

### 자습 복구 가능성 규칙
각 lesson 끝에는 `혼자 다시 따라오기` 섹션을 둔다. 이 섹션은 수업 중 놓친 학생을 위한 복구 경로다.

필수 포함 항목:
- 최소 재현 경로: 가장 작은 성공 시나리오
- 다시 볼 공식 문서와 읽을 키워드
- 스스로 확인할 명령 또는 화면
- 흔한 실패 3개와 첫 확인 위치
- 주의할 점과 다시 확인할 지점
- 다음 교시/다음 주차로 넘어가기 전 최소 준비 상태

## 강의 교안 디자인 생성 규칙
강의 교안은 처음 보는 학생도 천천히 따라올 수 있는 학습 자료 형태로 작성한다. 동시에 강의자가 자연스럽게 설명을 이어갈 수 있도록 흐름, 비유, 질문, 실습 절차, 시각 자료, 확인 기준을 충분히 포함한다.

### 학생용 학습자료 원칙
- 교안은 학생에게 그대로 배포될 수 있는 문서로 작성한다.
- "강의자 설명 스크립트", "강사가 말할 내용", "강사가 덧붙일 말"처럼 제작자 또는 강사용 메타 표현을 쓰지 않는다.
- 핵심 설명을 묶어야 할 때는 제작자 관점의 제목 대신 "핵심 포인트", "핵심 개념", "주의할 점"처럼 학생이 읽어도 자연스러운 제목을 쓴다.
- 강사만 참고해야 할 내부 지시문을 교안 본문에 넣지 않는다.
- 필요한 설명은 학생이 읽어도 자연스러운 서술형 설명으로 작성한다.
- 강의 진행 힌트가 필요하면 학습 흐름, 질문, 체크포인트, 정리 문장 안에 자연스럽게 녹인다.
- AI 생성 흔적으로 보일 수 있는 표현을 피하고, 공식 문서 기반 학습 자료처럼 작성한다.
- 인포그래픽, Mermaid, 표, 체크리스트가 설명을 자연스럽게 보완하도록 구성한다.

### 강의 산출물 폴더 구조
강의 산출물은 주차와 일차 단위로 명확하게 분리한다.

기본 구조:

```text
week{n}/
  README.md
  glossary.md
  day1/
    README.md
    lesson-01.md
    lesson-02.md
    assets/
  day2/
    README.md
    lesson-01.md
    lesson-02.md
    assets/
```

규칙:
- `week{n}`은 주차 번호를 의미한다. 예: `week1`, `week2`
- `day{n}`은 일차 번호를 의미한다. 예: `day1`, `day2`
- 각 주차에는 반드시 `week{n}/README.md`를 둔다.
- 각 주차에는 반드시 `week{n}/glossary.md`를 둔다.
- 각 일차에는 반드시 `week{n}/day{n}/README.md`를 둔다.
- 교시별 강의안은 `lesson-01.md`, `lesson-02.md`처럼 정렬 가능한 이름을 사용한다.
- 시각 자료는 해당 일차의 `assets/` 폴더에 둔다.
- 공통으로 재사용하는 이미지는 주차 공통 `week{n}/assets/`를 둘 수 있다.
- imagegen으로 생성한 이미지는 사용되는 일차의 `assets/` 아래에 저장하고, README 또는 교시 파일에는 이미지의 학습 목적과 파일 경로만 표기한다.

### 주차 README 규칙
`week{n}/README.md`에는 해당 주차의 전체 인덱스와 주차 Overview를 포함한다.

필수 포함 항목:
- 주차 제목
- 주차 목표
- 주차 키워드
- 주차 전체 Overview
- 일차별 수업 인덱스
- 주차 산출물
- 주차 실습 환경 준비물
- 공식 참고 자료 목록
- 주차별 glossary 링크
- 비용/보안 주의사항
- 다음 주차와의 연결

예시:

```markdown
# Week 2: Docker 기초와 실행 환경 표준화

## Overview
이번 주는 애플리케이션 실행 환경을 Docker 이미지와 컨테이너로 표준화하는 방법을 배운다.

## Learning Goals
- image와 container의 차이를 설명한다.
- Dockerfile로 이미지를 빌드한다.
- Docker Compose로 여러 컨테이너를 실행한다.

## Schedule Index
- Day 1: Docker 개념과 기본 명령어
- Day 2: Dockerfile과 이미지 빌드
- Day 3: 네트워크, 볼륨, 환경변수
- Day 4: Docker Compose
- Day 5: 통합 실습과 발표

## Deliverables
- Dockerfile
- compose.yaml
- 장애 분석 확인
```

### 일차 README 규칙
`week{n}/day{n}/README.md`에는 그 날 수업의 인덱스와 일별 Overview를 포함한다.

필수 포함 항목:
- 일차 제목
- 오늘의 Overview
- 오늘의 학습 목표
- 교시별 인덱스
- 오늘 사용할 공식 참고 자료
- 오늘의 핵심 용어
- 오늘 사용할 실습 파일과 assets
- 설치/계정/권한/비용 준비사항
- 오늘의 산출물
- 수업 종료 전 체크리스트

예시:

```markdown
# Week 2 Day 1: Docker 개념과 기본 명령어

## Overview
오늘은 Docker가 해결하려는 문제를 이해하고, 컨테이너를 실행하고 확인하고 정리하는 기본 사이클을 익힌다.

## Lesson Index
- 1교시: 1주차 복습 및 Docker 학습 목표
- 2교시: Docker의 컨셉과 작동 방식
- 3교시: Docker vs Local Computer
- 4교시: Docker 기본 명령어
- 5교시: Hello World와 nginx 실행
- 6교시: 표준 실습 앱 이미지 실행
- 7교시: 개인 면담 및 환경 점검
- 8교시: 보충 실습

## Checklist
- `docker version` 확인
- `docker run hello-world` 성공
- 표준 실습 앱 이미지 pull/run 성공
```

### 공통 디자인 원칙
- 주차, 일차, 교시 범위를 벗어나지 않는다.
- 한 교시는 하나의 핵심 문제와 하나의 핵심 메시지를 가진다.
- 처음 보는 학생도 따라할 수 있도록 설치, 계정 생성, 로그인, 버전 확인, 실행 확인 절차를 생략하지 않는다.
- 설명은 쉬운 말에서 시작하고, 공식 용어로 다시 연결한다.
- 핵심 용어는 처음 등장할 때 짧은 주석을 붙인다.
- 화면에 보여줄 자료와 서술형 설명이 자연스럽게 연결되도록 작성한다.
- 모든 교안에는 비용 절감, 개발/배포 효율성, 관리 효율성 중 적어도 하나의 관점을 명시한다.
- 개념만 설명하고 끝내지 않는다. 확인할 명령어, 콘솔 화면, 로그, 메트릭, 비용 화면 중 하나 이상을 포함한다.

### 강의 깊이 기준
강의안은 단순 목차나 요약 노트가 아니라 50분 수업을 실제로 버틸 수 있는 학습 자료여야 한다. 대학 강의 수준의 개념적 깊이와 현업 운영 관점을 함께 갖추되, 학생이 처음 봐도 따라갈 수 있는 단계성을 유지한다.

각 교시는 다음 조건을 만족해야 한다.

- 기본 요구사항보다 200% 확장한다는 기준으로, 필수 개념에는 정의/배경/운영 문제/한계/실패 모드/복구 절차/확인 지점을 모두 포함한다.
- 교안 첫 부분에 `오늘 반드시 가져갈 것`을 두고, 필수 개념 3~5개를 우선순위로 부각한다.
- 핵심 개념은 정의, 등장 배경, 해결하는 문제, 한계, 운영 판단 기준까지 설명한다.
- 실습은 "명령 실행"에서 끝나지 않고 준비, 실행, 관찰, 변경, 실패 재현, 복구, 확인까지 포함한다.
- 학생이 따라 칠 명령에는 기대 결과, 실패했을 때의 증상, 확인할 로그나 상태값을 함께 둔다.
- 적어도 하나의 의사결정 표를 포함해 "무엇을 보고 무엇을 바꿀지"를 판단하게 한다.
- 운영 관점의 질문을 포함한다. 예: "이 변경은 비용을 늘리는가?", "재현 가능한가?", "다른 사람이 같은 결과를 만들 수 있는가?", "장애가 나면 어디서 증거를 찾는가?"
- 개념 간 연결을 명시한다. 예: 배포는 Docker 이미지, Kubernetes rollout, AWS 배포, Terraform 재현성으로 이어진다.
- 피상적 표현을 금지한다. "중요하다", "편리하다", "쉽다"라고만 쓰지 말고 어떤 리스크와 비용을 줄이는지 설명한다.
- 한 교시의 본문은 표, 명령, Mermaid, 확인 양식을 제외하고도 충분한 서술형 설명을 포함한다. 짧은 bullet 목록만으로 교안을 끝내지 않는다.
- 각 교시 끝에는 `혼자 다시 따라오기` 섹션을 두어, 수업 중 놓친 학생이 최소 성공 경로와 흔한 실패 복구 지점을 확인할 수 있게 한다.
- 교수자 수준의 인사이트를 넣되, 학생에게 보이는 문서이므로 내부 강의 대본이나 제작 지시문처럼 보이는 문장은 쓰지 않는다.

### 시각화 필수 규칙
각 lesson 본문에는 시각자료를 2~3개 포함한다. README나 일차 개요에만 있는 이미지는 lesson의 시각자료로 인정하지 않는다. 시각자료는 학습 목적이 분명해야 하며, 이미지 아래에는 반드시 "무엇을 봐야 하는지"와 "이 visual이 어떤 개념을 설명하는지"를 적는다.

imagegen으로 생성한 교안용 인포그래픽, 공식 문서/공식 블로그/대학 강의자료 이미지, Mermaid 다이어그램, 터미널/브라우저 화면 캡처 가이드 중에서 lesson 성격에 맞게 조합한다. 공식 이미지를 가져올 수 있으면 출처를 표기해 사용하고, 가져오기 어렵거나 수업 맥락에 맞는 그림이 없으면 imagegen으로 직접 만든다.

필수 시각화 유형:
- 공식 이미지 또는 공식 다이어그램
- 직접 생성한 인포그래픽
- Mermaid 다이어그램
- 콘솔/터미널/대시보드 화면 캡처 가이드
- 아키텍처 흐름도
- 장애 분석 플로우차트

시각화 사용 기준:
- 공식 문서에 적절한 이미지, 구조도, 아키텍처 다이어그램이 있으면 우선 사용한다.
- 공식 이미지를 사용할 때는 출처 링크와 문서 제목을 함께 표기한다.
- 공식 이미지의 라이선스, 로고 사용 규칙, 수정 가능 여부를 확인한다.
- 공식 이미지가 있더라도, 각 강의마다 학생 눈높이에 맞춘 imagegen 인포그래픽을 별도로 생성한다.
- 단순 구조, 흐름, 의존성, 상태 전이는 Mermaid를 우선 사용한다.
- 추상 개념, 비유, 시각적 기억이 필요한 설명은 인포그래픽을 사용한다.
- 생성형 이미지가 필요한 경우 `imagegen` 스킬을 사용해 교안용 인포그래픽을 만든다.

### 학생용 이미지 톤 규칙
이미지는 운영 시간표가 아니라 학생이 개념을 기억하고 복습하는 학습 자산이어야 한다. 시간, 교시, 분 단위 운영 정보는 lesson 본문이나 README 표에만 둔다. PNG/WebP/JPG asset에는 가능한 한 넣지 않는다.

이미지에 넣지 않는 요소:
- `1교시`, `2교시`, `Lesson 1`, `Day 1 Lesson 2`처럼 교안 인덱스로 보이는 제목
- `0-10분`, `50분 흐름`, `10분 휴식`처럼 시간표로 보이는 문구
- 강사용 진행 순서, 운영 메모, 발표 시간 배분, 체크인/휴식 시간
- 표지만 바꾼 일정표, 세션 목록, 목차형 이미지

이미지에 넣는 요소:
- 개념 간 관계, 실행 흐름, 장애 흐름, 확인 지점 위치, 판단 기준
- 학생이 복습할 때 바로 떠올릴 수 있는 짧은 핵심어
- 주차/일차 표기는 필요한 경우에만 작게 넣고, 주된 제목은 기술 개념이나 운영 문제로 둔다.
- 시간 순서가 필요하면 `준비 -> 실행 -> 관찰 -> 복구 -> 확인`처럼 학습/운영 단계로 표현한다.

예를 들어 `50분 흐름` 이미지를 만들지 말고 `장애를 좁히는 관찰 흐름`, `Compose 실행 조건 지도`, `서비스 연결 실패 분석 경로`처럼 개념 중심 제목을 사용한다. 이미지를 보고 "수업 계획표"처럼 느껴지면 실패다.

### OT/입문 세션 작성 규칙
OT는 기술 수업이 아니라 학습 동선을 안정화하는 시간이다.

- 공개 질문을 많이 던지지 않는다. 한 세션에서 전체 토론 질문은 1~2개로 제한하고, 나머지는 개인 작성 카드나 체크리스트로 바꾼다.
- "오늘 하지 않는다", "아직 다루지 않는다" 같은 부정형 범위 문장을 반복하지 않는다. 필요한 경우 "오늘의 초점은 운영 안내와 학습 준비"처럼 긍정형으로 표현한다.
- 학생이 첫날 기억해야 할 것은 시간, 장소, 소통 경로, 산출물, 안전 원칙, 학습 로드맵이다.
- AI coding agent 시대 인사이트는 반드시 포함하되, 겁주거나 도구를 홍보하는 방식이 아니라 "작은 앱 생성"과 "business/service 운영" 사이에 필요한 인프라 책임을 설명하는 방식으로 다룬다.

### 공식 이미지와 생성 이미지 사용 원칙
- 공식 documentation, 공식 블로그, 공식 발표 자료의 이미지를 우선 검토한다.
- 공식 이미지가 너무 복잡하면 원본 링크를 제공하고, 학생용 단순화 인포그래픽을 별도로 만든다.
- 공식 로고나 제품 UI를 임의로 왜곡하지 않는다.
- 각 강의마다 imagegen으로 생성한 이해 보조 인포그래픽을 반드시 포함한다.
- 생성 이미지는 공식 다이어그램을 대체하는 권위 자료가 아니라 이해 보조 자료로 사용한다.
- 생성 이미지에는 실제 서비스 UI나 비용 수치처럼 정확성이 중요한 내용을 넣지 않는다. 정확한 값과 UI 절차는 공식 문서, 콘솔 화면, 텍스트 절차로 제공한다.
- imagegen으로 만든 이미지는 프로젝트 내 assets 경로에 저장하고, 교안에는 파일 경로와 생성 목적만 표기한다.

AWS 서비스, Kubernetes 리소스, Helm chart, Terraform provider/resource, Observability 도구처럼 특정 서비스나 컴포넌트를 다룰 때는 반드시 공식 문서를 기준으로 설명한다.

필수 포함 항목:
- 공식 documentation 링크
- 공식 documentation에서 확인한 핵심 정의
- 주요 설정 항목
- 기본값
- 제약 스펙
- 제약점과 운영상 주의사항
- 비용 발생 조건
- 보안상 주의사항
- 확장 또는 한계에 도달했을 때의 선택지

### 핵심 지표 누락 방지 규칙
각 주제에서 현업 판단에 필요한 핵심 지표, 용어, 의사결정 기준이 빠지지 않도록 한다. 도구 사용법만 설명하고 판단 기준을 생략하면 안 된다.

주제별 필수 검토 지표 예시:
- 클라우드 전환: CAPEX, OPEX, TCO, ROI, 사용량 기반 과금, 감가상각, 리소스 폐기 비용
- DevOps: deployment frequency, lead time for changes, change failure rate, time to restore service
- Docker: image size, build time, startup time, vulnerability count, resource usage
- MSA: latency, error rate, dependency count, blast radius, timeout, retry count
- Kubernetes: desired replicas, available replicas, restart count, CPU/Memory requests and limits, node utilization
- Observability: logs, metrics, traces, SLI, SLO, error budget, alert fatigue
- FinOps: budget, forecast, actual cost, unit cost, idle resource, tag coverage
- Terraform/IaC: plan diff, drift, state risk, resource count, destroy impact

작성 기준:
- 지표는 정의만 쓰지 말고 왜 의사결정에 필요한지 설명한다.
- 초급자에게 어려운 지표는 쉬운 비유와 인포그래픽으로 연결한다.
- 공식 문서나 공식 프레임워크에서 쓰는 용어가 있으면 그 용어를 우선 사용한다.
- 해당 교시 범위에서 깊게 다루지 않을 지표는 "오늘은 개념만 소개하고, 이후 N주차에서 실습한다"처럼 위치를 명확히 한다.

### 계산 예제 필수 규칙
비용, 용량, 성능, 확장성, 운영 인력, 장애 복구 시간처럼 실제 판단에 숫자가 필요한 주제는 반드시 단순 계산 예제를 포함한다.

계산 예제가 필요한 대표 주제:
- 클라우드 전환: 데이터센터 CAPEX/OPEX/TCO와 클라우드 사용량 기반 비용 비교
- FinOps: 월 비용, 일 비용, 시간 비용, idle resource 비용, 태그별 비용
- Kubernetes: replica 수, node 수, CPU/Memory requests, autoscaling 전후 비용
- Observability: 로그 수집량, 보관 기간, 알림 빈도, 대시보드 지표 수
- Terraform/IaC: 생성 리소스 수, destroy 영향 범위, drift 발생 리소스 수
- DevOps: 배포 리드 타임, 장애 복구 시간, 변경 실패율

계산 예제 작성 기준:
- 실제 가격은 변동되므로 공식 pricing 문서 또는 계산기 링크를 함께 제공한다.
- 숫자는 교육용 가정값임을 명시하고, 환율과 기준 날짜를 함께 적는다.
- 계산식은 학생이 손으로 따라갈 수 있을 만큼 단순하게 쓴다.
- 단기 사용과 장기 사용을 비교한다.
- 직접 구매 비용뿐 아니라 교체 주기, 건물/상면, 전력, 냉각, 네트워크, 운영 인력, 장애 대응 비용을 함께 고려한다.
- 클라우드 비용은 비즈니스 규모, 아키텍처 타입, 사용 시간, 트래픽 패턴, 관리형 서비스 선택에 따라 언제든지 달라질 수 있음을 명시한다.
- 스몰 비즈니스의 사내 전용 도구처럼 요구사항이 작고 고정적이면 로컬 컴퓨터나 기존 장비가 더 경제적일 수 있음을 함께 설명한다.
- 대규모 아키텍처라도 POC 수준에서 몇 시간만 만들고 종료한다면 클라우드가 초기 투자 없이 빠르게 검증하는 데 유리할 수 있음을 설명한다.
- 클라우드는 규모의 경제, 사용량 기반 과금, 관리형 서비스로 인해 짧은 실험과 변동 워크로드에 유리할 수 있음을 설명한다.
- 반대로 오래 지속되고 사용량이 예측 가능하며 대규모인 워크로드는 데이터센터 또는 장기 약정/예약 모델이 유리할 수 있음을 함께 설명한다.

### 핵심 용어 주석과 주차별 Glossary 규칙
핵심 단위 용어는 학생이 나중에 쉽게 복습하고 암기할 수 있도록 교안 안에서 짧게 주석화하고, 주차별 `glossary.md`에 누적한다.

용어 주석 작성 기준:
- 용어가 처음 등장하는 위치에 `용어(짧은 뜻)` 형식으로 적는다.
- 영어 약어는 풀네임과 한국어 의미를 함께 적는다.
- 한 문장 안에 긴 정의를 넣지 않는다. 자세한 설명은 glossary로 보낸다.
- 같은 교안에서 같은 용어를 반복해서 길게 설명하지 않는다.
- 공식 문서에서 쓰는 원어를 유지하되, 학생이 이해할 수 있는 한국어 뜻을 붙인다.

예시:
- `CAPEX(Capital Expenditure, 초기 투자 비용)`
- `OPEX(Operational Expenditure, 운영 비용)`
- `TCO(Total Cost of Ownership, 총소유비용)`
- `Pod(Kubernetes에서 함께 배치되는 컨테이너 실행 단위)`
- `Security Group(AWS 인스턴스 단위 방화벽)`

주차별 glossary 작성 기준:
- 파일 위치는 `week{n}/glossary.md`로 고정한다.
- 용어는 카테고리별로 묶는다. 예: Cloud, DevOps, Linux/CLI, Git, Cost, Security
- 각 용어는 다음 형식을 사용한다.

```markdown
## Category

### Term
- 한 줄 뜻:
- 왜 중요한가:
- 수업에서 다시 나오는 곳:
- 공식 참고:
```

암기 보조 규칙:
- 비슷한 용어는 함께 비교한다. 예: CAPEX vs OPEX, image vs container, liveness vs readiness
- "한 줄 뜻"은 20~40자 안팎으로 짧게 쓴다.
- 용어가 실제 실습에서 어떻게 확인되는지 명령어, 콘솔 화면, 파일명과 연결한다.
- 일차 README에는 그날 처음 등장하는 핵심 용어 5~15개를 `오늘의 핵심 용어`로 요약한다.

제약 스펙 작성 기준:
- 숫자로 표현 가능한 제한은 숫자로 적는다.
- 리전, 버전, 플랜, 인스턴스 타입, 클러스터 종류에 따라 달라지는 제한은 전제 조건을 함께 적는다.
- 공식 문서에서 확인하지 못한 제한은 단정하지 않는다.
- 불확실한 내용은 "공식 문서 확인 필요"로 표시한다.

제약점 작성 기준:
- "무엇을 할 수 없다"만 쓰지 말고, 운영에서 어떤 문제가 되는지 설명한다.
- 우회 방법이 있으면 함께 적는다.
- 우회 방법이 비용, 복잡도, 보안 위험을 증가시키면 그 점도 적는다.

예시:
- ALB는 외부 트래픽을 여러 target으로 분산하지만, target group health check가 실패하면 정상 인스턴스가 있어도 트래픽을 받지 못할 수 있다.
- RDS는 운영 부담을 줄여주지만, 엔진별 버전, storage, backup, Multi-AZ, 삭제 보호 설정에 따라 비용과 제약이 달라진다.
- Kubernetes HPA는 부하에 따라 replica를 늘릴 수 있지만, metrics-server나 적절한 resource requests가 없으면 기대대로 동작하지 않는다.
- Terraform state는 현재 인프라를 추적하지만, 민감정보가 포함될 수 있고 여러 사람이 동시에 수정하면 충돌이 생길 수 있다.

### 외부 사례와 발표 자료 사용 규칙
서비스별 컴포넌트를 설명할 때는 가능한 한 실제 사용 사례를 함께 보여준다. 학생이 "이걸 왜 배우는지"를 현업 사례로 연결하기 위함이다.

자료 우선순위:
1. 국내 기업의 공식 기술 블로그, 컨퍼런스 발표, 공개 세미나 자료
2. 국내 클라우드/DevOps/Kubernetes 커뮤니티 발표 자료
3. AWS Summit Seoul, Google Cloud Summit Seoul, Naver, Kakao, Toss, 당근, 우아한형제들, 쿠팡, LINE 등 국내 또는 국내 대상 공식 발표 자료
4. 국내 자료가 부족하면 글로벌 공식 발표 자료
5. 그래도 부족하면 벤더 공식 블로그 또는 공식 case study

사례 작성 기준:
- 발표 제목 또는 문서 제목
- 발표/게시 주체
- 링크
- 이 강의 주제와 연결되는 지점
- 학생에게 보여줄 핵심 장면 또는 핵심 메시지
- 그대로 따라 하면 안 되는 이유 또는 우리 실습과 다른 점

주의:
- 외부 사례는 권위 부여용 장식이 아니라 수업 개념을 현실 문제와 연결하는 용도로 사용한다.
- 사례가 오래된 경우 현재 공식 문서와 비교해 달라진 점을 확인한다.
- 국내 발표 자료를 우선하되, 공식 문서와 충돌하면 공식 문서를 우선한다.

### 50분 강의 분량 규칙
각 교시 교안은 실제 50분 동안 설명과 실습이 가능할 만큼 충분히 길어야 한다. 짧은 요약이나 슬라이드 제목 수준으로 작성하지 않는다.

권장 시간 배분:
- 0~5분: 시작 질문과 문제 상황 제시
- 5~15분: 핵심 개념과 비유 설명
- 15~25분: 공식 문서 기반 스펙, 제약, 사례 설명
- 25~40분: 실습 또는 데모
- 40~47분: 결과 확인, 흔한 오류, 장애/비용/보안 포인트
- 47~50분: 정리와 다음 교시 연결

교안 분량 기준:
- 개념형 교안은 학생이 읽어도 자연스러운 서술형 설명과 사례 설명을 충분히 포함한다.
- 핸즈온형 교안은 설치, 실행, 검증, 오류 대응, 정리 절차를 빠짐없이 포함한다.
- 장애 분석형 교안은 재현, 관찰, 가설, 검증, 복구, 재발 방지까지 포함한다.
- 각 교시는 최소 50분 강의 흐름을 채울 수 있도록 세부 설명, 질문, 예시, 실습 체크포인트를 포함한다.
- 단순 bullet list만으로 교안을 끝내지 않는다.

### 비유와 인포그래픽 규칙
개념 강의에는 반드시 비유를 포함하고, 비유와 연결되는 인포그래픽도 포함한다. 비유를 사용한 모든 교안에는 해당 비유를 시각화한 imagegen 인포그래픽이 필수다.

비유 작성 기준:
- 학생이 이미 아는 생활 상황으로 시작한다.
- 비유가 설명하는 기술 개념을 바로 연결한다.
- 비유의 한계를 명시한다.
- 비유가 과장되어 기술 개념을 왜곡하지 않게 한다.

비유 인포그래픽 작성 기준:
- 한 장의 그림에서 생활 비유와 기술 개념이 나란히 대응되게 만든다.
- 예: "항구와 물류"를 Docker image/container/registry에 대응
- 예: "건물 출입문"을 port/security group/ingress에 대응
- 예: "배전반"을 observability/runbook에 대응
- 교안의 비유 섹션 바로 근처에 인포그래픽을 배치한다.
- 이미지 설명에는 이 인포그래픽이 어떤 비유와 어떤 기술 개념을 연결하는지 적는다.
- 단순 장식용 이미지는 허용하지 않는다. 반드시 학습 개념과 대응 관계가 있어야 한다.

### Mermaid 사용 규칙
Mermaid는 재현 가능하고 수정 쉬운 시각 자료가 필요할 때 사용한다.

우선 사용 대상:
- 요청 흐름
- 서비스 의존성
- 배포 단계
- 장애 분석 순서
- Terraform plan/apply/destroy 흐름
- AWS 리소스 연결 구조
- Kubernetes 리소스 관계

Mermaid 작성 기준:
- 한 다이어그램에는 핵심 노드만 넣는다.
- 학생이 처음 보는 약어는 노드명에 풀어서 쓴다.
- 너무 복잡해지면 overview와 detail 두 장으로 나눈다.
- Mermaid 옆에는 반드시 다이어그램을 읽는 순서를 설명한다.

## DevOps 절대 원칙

### 1. 비용 절감 원칙
모든 인프라 선택은 비용을 만든다. 비용은 나중에 보는 것이 아니라 설계 순간부터 함께 봐야 한다.

강의에서는 다음 질문을 반복한다.
- 이 리소스는 왜 필요한가?
- 더 작은 리소스로 시작할 수 있는가?
- 사용하지 않을 때 꺼지거나 삭제되는가?
- 비용이 발생하는 단위는 무엇인가?
- 누가 비용을 확인하고 책임지는가?

예시:
- EC2는 켜져 있는 시간만큼 비용이 발생한다.
- RDS는 만들기 쉽지만, 백업, 스토리지, Multi-AZ 설정에 따라 비용이 커진다.
- Kubernetes는 Pod만 보는 것이 아니라 node 비용과 plugin 운영 비용까지 봐야 한다.
- 로그와 메트릭도 저장 기간과 수집량에 따라 비용이 된다.

비유:
- 클라우드 리소스는 전기 스위치가 아니라 수도꼭지에 가깝다. 조금 열어도 계속 흐르면 요금이 나온다.

### 2. 개발 및 배포 효율성 원칙
DevOps의 핵심은 개발자가 만든 코드를 더 빠르고 안정적으로 사용자에게 전달하는 것이다.

강의에서는 다음 질문을 반복한다.
- 배포 과정에서 사람이 반복해서 클릭하고 있는가?
- 같은 배포를 다시 재현할 수 있는가?
- 실패했을 때 어디서 실패했는지 확인할 수 있는가?
- rollback 또는 복구 경로가 있는가?
- 개발팀이 인프라팀에 요청할 때 필요한 정보가 명확한가?

예시:
- Dockerfile은 실행 환경을 표준화한다.
- Docker Compose는 여러 컨테이너 실행 절차를 파일로 남긴다.
- Kubernetes Deployment는 원하는 Pod 수와 배포 상태를 선언한다.
- Terraform은 콘솔 클릭 절차를 코드와 리뷰 가능한 변경으로 바꾼다.

비유:
- 수동 배포는 매번 다른 사람이 라면을 끓이는 것과 같다. Terraform과 Kubernetes는 레시피를 남겨 같은 결과를 반복하게 한다.

### 3. 관리 효율성 원칙
운영 가능한 시스템은 사람이 머리로 기억하지 않아도 상태를 확인하고 변경할 수 있어야 한다.

강의에서는 다음 질문을 반복한다.
- 현재 상태를 어디서 확인하는가?
- 설정값은 어디에 있는가?
- 장애가 나면 어떤 순서로 확인하는가?
- 누가 보아도 같은 방식으로 운영할 수 있는가?
- 문서, 코드, 대시보드가 실제 상태와 맞는가?

예시:
- README에는 실행 명령과 확인 명령이 있어야 한다.
- Kubernetes manifest는 현재 배포 의도를 문서처럼 보여준다.
- CloudWatch와 Grafana는 사람이 계속 터미널을 보지 않아도 상태를 확인하게 한다.
- Terraform state와 plan은 현재 인프라와 변경 예정 상태를 비교하게 한다.

비유:
- 운영 문서가 없는 인프라는 라벨 없는 배전반과 같다. 불이 꺼졌을 때 어떤 스위치를 봐야 할지 알 수 없다.

## 강의 구성 원칙

각 교시는 가능하면 다음 흐름을 따른다.

1. 문제 제시
   - 학생이 이미 겪었거나 곧 겪을 운영 문제를 먼저 보여준다.
   - 예: "내 PC에서는 되는데 다른 사람 PC에서는 왜 안 될까?"

2. 개념 설명
   - 문제를 해결하기 위해 등장한 개념을 설명한다.
   - 용어 정의만 하지 말고, 어떤 불편함을 줄이는지 연결한다.

3. 비유
   - 추상 개념은 생활 비유로 먼저 잡고, 곧바로 기술 개념으로 되돌아온다.
   - 비유는 이해를 돕는 도구이며, 비유 자체가 결론이 되면 안 된다.

4. 실습
   - 학생이 직접 만들고 실행하고 확인한다.
   - 실습은 항상 실행, 확인, 변경, 장애 확인, 정리 중 하나 이상을 포함한다.

5. 관찰
   - 명령어 결과, 로그, 메트릭, 콘솔 상태, 비용 화면 중 하나를 반드시 확인한다.
   - "실행됐다"에서 끝내지 않고 "어떤 증거로 실행됐다고 판단하는가"를 묻는다.

6. 정리
   - 오늘 배운 개념이 비용, 배포 효율성, 관리 효율성 중 어디에 연결되는지 정리한다.
   - 다음 교시 또는 다음 주차와 연결한다.

## 강의안 작성 템플릿

각 교시 강의안은 아래 구조를 기본으로 한다.

```markdown
## N교시 : 제목

### 수업 목표
- 학생이 수업 후 설명하거나 수행할 수 있어야 하는 것

### 시작 질문
- 학생의 경험이나 운영 문제를 끌어내는 질문

### 핵심 개념
- 반드시 설명해야 할 개념 3~5개

### 공식 참고 자료
- 공식 documentation, 공식 블로그, 글로벌 발표 자료 링크
- 학생이 문서에서 직접 확인해야 할 키워드

### 비유
- 개념 이해를 돕는 짧은 비유

### 강의 흐름
1. 문제 상황 제시
2. 개념 설명
3. 실습 또는 데모
4. 결과 확인
5. 장애 또는 실수 포인트 확인
6. 정리

### 실습
- 명령 또는 콘솔 절차
- 확인해야 할 결과

### 이해 확인 질문
- 왜 이 방식이 필요한가?
- 비용은 어디서 발생하는가?
- 배포 효율성은 어떻게 좋아지는가?
- 관리 효율성은 어떻게 좋아지는가?
- 공식 문서에서는 이 개념을 어떻게 설명하는가?

### 흔한 실수
- 학생들이 자주 놓치는 지점

### 마무리 정리
- 이 교시가 다음 교시와 어떻게 연결되는지
```

## 주차별 강의 생성 방향

### 1주차: Cloud Native 기본기와 운영 마인드셋
초점은 도구 사용이 아니라 운영 가능한 서비스를 바라보는 눈을 만드는 것이다.

강조할 관점:
- 컴퓨터, 네트워크, 스토리지, 프로세스, 포트, 로그는 모든 클라우드 기술의 바닥이다.
- AI는 학습 보조 도구이지 책임을 대신지는 도구가 아니다.
- GitHub, README, 실행 명령, 장애 확인은 협업의 기본이다.

대표 비유:
- 포트는 건물의 출입문 번호와 같다.
- 로그는 CCTV와 출입 확인의 조합이다.
- README 없는 프로젝트는 조립 설명서 없는 가구와 같다.

### 2주차: Docker 기초와 실행 환경 표준화
초점은 Docker 명령어가 아니라 "실행 환경을 어떻게 포장하고 재현할 것인가"이다.

강조할 관점:
- 이미지는 실행 재료이고, 컨테이너는 실행 중인 프로세스다.
- Dockerfile은 실행 환경 레시피다.
- Docker Hub는 이미지를 공유하는 물류 창고다.
- Compose는 여러 컨테이너 실행 절차를 문서화한 운영 리모컨이다.

대표 비유:
- 이미지는 도시락 레시피와 포장된 도시락 사이에 가깝고, 컨테이너는 실제로 먹기 위해 열린 도시락이다.
- 볼륨은 컨테이너가 사라져도 남겨야 하는 외부 보관함이다.

### 3주차: 인프라 관점의 MSA
초점은 서비스를 개발하는 방법이 아니라 여러 서비스로 나뉜 시스템을 운영하는 방법이다.

강조할 관점:
- MSA는 운영해야 할 대상이 늘어나는 구조다.
- 서비스 목록, 포트, 프로토콜, 의존성, 헬스체크, 로그 위치가 명확해야 한다.
- 개발팀에 요구할 정보와 인프라팀이 보장할 정보를 구분해야 한다.
- MSA는 정답이 아니라 트레이드오프다.

대표 비유:
- Monolith는 한 건물 안에 모든 부서가 있는 회사이고, MSA는 여러 지점으로 나뉜 회사다. 지점이 늘면 전문성은 좋아질 수 있지만 연락망과 장애 대응이 어려워진다.

### 4주차: Kubernetes와 운영 확장 컴포넌트
초점은 YAML 암기가 아니라 여러 서버에서 서비스를 안정적으로 운영하는 선언적 모델을 이해하는 것이다.

강조할 관점:
- Kubernetes는 "원하는 상태"를 선언하고 그 상태에 맞추려는 시스템이다.
- multi-node 로컬 클러스터는 여러 서버 운영 환경을 작게 재현하는 실습장이다.
- Helm은 Kubernetes 플러그인을 설치하고 관리하는 패키지 매니저다.
- Ingress, autoscaler, Istio, Grafana stack은 운영 요구사항이 커질 때 등장한다.

대표 비유:
- Kubernetes는 개별 직원을 직접 지시하는 관리자가 아니라, 조직도를 보고 빈자리를 채우는 운영 관리자다.
- Helm chart는 Kubernetes용 설치 설명서와 설치 패키지를 묶은 것이다.

### 5주차: AWS Console, FinOps, Observability, Terraform/IaC
초점은 AWS 서비스 이름을 외우는 것이 아니라 컴퓨팅 구성요소가 클라우드 서비스로 어떻게 확장되는지 이해하는 것이다.

강조할 관점:
- EC2, VPC, S3, RDS, ALB, CloudWatch는 각각 기존 컴퓨팅 구성요소의 클라우드 버전이다.
- 콘솔 구성은 배우기 쉽지만 반복, 리뷰, 재현이 어렵다.
- 비용과 관찰 가능성은 아키텍처 선택의 일부다.
- 리소스를 만드는 것만큼 정리하는 것도 실습이다.
- Terraform plan은 변경 전 견적서이자 작업 예고서다.
- state는 Terraform이 기억하는 현재 인프라 장부다.
- drift는 실제 인프라와 코드의 불일치다.
- IaC에도 비용, 보안, 관찰 가능성 기준이 들어가야 한다.

대표 비유:
- AWS Console은 공장 설비를 직접 손으로 배치하는 작업장이다. 처음 배우기는 좋지만, 같은 공장을 여러 번 만들려면 설계도가 필요하다.
- Terraform은 인프라 시공 도면과 작업 명세서다. 콘솔 클릭이 현장 작업자의 기억에 의존한다면, Terraform은 변경 내용을 문서와 코드로 남긴다.

## 강의 생성 시 금지할 흐름
- 개발 프레임워크 내부 구현 설명에 과도하게 들어가지 않는다.
- "이 명령어를 외우세요" 식으로 끝내지 않는다.
- 비용이 발생하는 실습에서 삭제 절차를 생략하지 않는다.
- 정상 실행만 보여주고 장애 상황을 다루지 않는 강의를 만들지 않는다.
- AI가 생성한 답을 검증 없이 그대로 쓰는 흐름을 만들지 않는다.
- MSA, Kubernetes, Terraform을 무조건 좋은 기술처럼 설명하지 않는다.
